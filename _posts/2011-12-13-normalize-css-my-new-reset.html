Many of us who work with CSS on a daily basis have some kind of reset mechanism built in. There are a bunch of options out there these days, but for many the timeless classic is Eric Meyer's CSS reset. Another popular offering in today's HTML5 wilds is the HTML5 doctor HTML5 CSS reset.

Is all that really necessary?

We use these resets to bring every browser to a common ground. Wipe out padding, margin, font-size variations, and the like. We forget that browsers have some defaults that can be handy. Margin on block elements can be very helpful. Why change something you don't need to. Another useful convention implemented by most browsers are greater size/weight given to header tags. This is usually desired.

Enter normalize.css

Why not get the best of both worlds? I have been using this bad motha in projects lately, and I am pleased with what I see. According to the normalize.css site normalize does the following:
  • Preserves useful defaults, unlike many CSS resets
  • Normalizes styles for a wide range of elements
  • Corrects bugs and common browser inconsistencies.
  • Improves usability with subtle improvements
  • Explains what code does using detailed comments.
You get a lot of bonuses you might otherwise take for granted. Things like audio elements without controls (which are useless) being hidden, maintaining consistent font size with ems, cake sure pre-formatted text is readable, consistent sup and sub elements, image quality when scaling in IE, and a tone more. Give it a look-see!

One more thing....
